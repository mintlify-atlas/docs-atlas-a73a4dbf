## Overview

The `traceability-graph.json` file is a structured representation of all SDD artifacts, their relationships, code/test/commit references, and statistical metrics. It powers the interactive HTML dashboard and the MCP server.

## Location

`dashboard/traceability-graph.json`

## Schema version

**Current:** v4 (introduced in SDD Plugin v2.0.0)

**Backward compatibility:** All v1/v2/v3 fields are preserved. Each version is a pure extension.

## Root fields

<ResponseField name="$schema" type="string" required>
  Schema version identifier. Always `"traceability-graph-v4"`.
</ResponseField>

<ResponseField name="generatedAt" type="string" required>
  ISO-8601 timestamp when the graph was generated.
  
  Example: `"2026-03-01T15:30:00.000Z"`
</ResponseField>

<ResponseField name="projectName" type="string" required>
  Name of the project. Derived from `package.json`, directory name, or `pipeline-state.json`.
</ResponseField>

<ResponseField name="pipeline" type="object" required>
  Pipeline execution status snapshot.
  
  <Expandable title="Pipeline object">
    <ResponseField name="currentStage" type="string" required>
      Active stage name from `pipeline-state.json`, or `"unknown"` if file not found.
    </ResponseField>
    
    <ResponseField name="stages" type="array" required>
      Ordered array of 7 pipeline stages with their status.
      
      <Expandable title="Stage object">
        <ResponseField name="name" type="string" required>
          Stage identifier (e.g., `"requirements-engineer"`).
        </ResponseField>
        
        <ResponseField name="status" type="enum" required>
          One of: `"done"`, `"stale"`, `"running"`, `"error"`, `"pending"`.
        </ResponseField>
        
        <ResponseField name="lastRun" type="string | null" required>
          ISO-8601 timestamp of last completion. `null` if never run.
        </ResponseField>
        
        <ResponseField name="artifactCount" type="number" required>
          Count of artifacts produced by this stage.
        </ResponseField>
      </Expandable>
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="artifacts" type="array" required>
  Array of all SDD artifacts found in the project.
  
  <Expandable title="Artifact object">
    <ResponseField name="id" type="string" required>
      Unique artifact identifier. Format depends on type:
      - `REQ-{PREFIX}-{NNN}` (e.g., `REQ-EXT-001`)
      - `UC-{NNN}`, `WF-{NNN}`, `BDD-{NNN}`
      - `INV-{PREFIX}-{NNN}`, `API-{name}`, `ADR-{NNN}`
      - `TASK-F{N}-{NNN}`, `FASE-{N}`
    </ResponseField>
    
    <ResponseField name="type" type="string" required>
      Artifact type: `REQ`, `UC`, `WF`, `API`, `BDD`, `INV`, `ADR`, `NFR`, `RN`, `FASE`, `TASK`.
    </ResponseField>
    
    <ResponseField name="category" type="string | null">
      Sub-category (e.g., `EXT`, `CVA`, `SYS`, `SEC`, `F`, `NF`). `null` if not applicable.
    </ResponseField>
    
    <ResponseField name="title" type="string" required>
      Artifact title or first line of definition.
    </ResponseField>
    
    <ResponseField name="file" type="string" required>
      Relative file path where the artifact is defined (forward slashes).
    </ResponseField>
    
    <ResponseField name="line" type="number" required>
      Line number of definition (1-indexed).
    </ResponseField>
    
    <ResponseField name="priority" type="string | null">
      Priority level if available (e.g., `"Must Have"`, `"Should Have"`).
    </ResponseField>
    
    <ResponseField name="stage" type="string" required>
      Pipeline stage that owns this artifact.
    </ResponseField>
    
    <ResponseField name="classification" type="object | null">
      Business/technical/functional classification. Only present for REQ artifacts.
      
      <Expandable title="Classification object">
        <ResponseField name="businessDomain" type="string" required>
          Business domain inferred from REQ prefix (e.g., `"Extraction & Processing"`).
        </ResponseField>
        
        <ResponseField name="technicalLayer" type="string" required>
          One of: `"Infrastructure"`, `"Backend"`, `"Frontend"`, `"Integration/Deployment"`, `"Unknown"`.
        </ResponseField>
        
        <ResponseField name="functionalCategory" type="string" required>
          One of: `"Functional"`, `"Non-Functional"`, `"Security"`, `"Data"`, `"Integration"`.
        </ResponseField>
      </Expandable>
    </ResponseField>
    
    <ResponseField name="codeRefs" type="array">
      Source code references implementing this artifact. Empty array `[]` if none.
      
      <Expandable title="CodeRef object">
        <ResponseField name="file" type="string" required>
          Relative path to source file.
        </ResponseField>
        
        <ResponseField name="line" type="number" required>
          Line number where the `Refs:` comment was found.
        </ResponseField>
        
        <ResponseField name="symbol" type="string" required>
          Nearest symbol name (function, class, const) or `"filename:line"` fallback.
        </ResponseField>
        
        <ResponseField name="symbolType" type="string" required>
          One of: `"function"`, `"class"`, `"const"`, `"interface"`, `"type"`, `"method"`, `"variable"`, `"unknown"`.
        </ResponseField>
        
        <ResponseField name="refIds" type="array" required>
          Artifact IDs referenced in the comment (e.g., `["UC-001", "INV-EXT-005"]`).
        </ResponseField>
        
        <ResponseField name="inferred" type="boolean">
          **v4 only.** `true` if this reference was inferred transitively via call graph. Absent in v1-v3.
        </ResponseField>
        
        <ResponseField name="confidence" type="number">
          **v4 only.** Confidence score 0.0-1.0 for inferred references. Absent in v1-v3.
        </ResponseField>
      </Expandable>
    </ResponseField>
    
    <ResponseField name="testRefs" type="array">
      Test references verifying this artifact. Empty array `[]` if none.
      
      <Expandable title="TestRef object">
        <ResponseField name="file" type="string" required>
          Relative path to test file.
        </ResponseField>
        
        <ResponseField name="line" type="number" required>
          Line number of the test definition or `Refs:` comment.
        </ResponseField>
        
        <ResponseField name="testName" type="string" required>
          Test description (e.g., `"validates size per INV-EXT-005"`).
        </ResponseField>
        
        <ResponseField name="framework" type="string" required>
          Test framework: `"vitest"`, `"jest"`, `"pytest"`, `"jasmine"`, `"unknown"`.
        </ResponseField>
        
        <ResponseField name="refIds" type="array" required>
          Artifact IDs referenced in the test.
        </ResponseField>
      </Expandable>
    </ResponseField>
    
    <ResponseField name="commitRefs" type="array">
      Git commits referencing this artifact via `Refs:`/`Task:` trailers. Empty array `[]` if none.
      
      <Expandable title="CommitRef object">
        <ResponseField name="sha" type="string" required>
          Short SHA (7 chars) of the commit.
        </ResponseField>
        
        <ResponseField name="fullSha" type="string" required>
          Full 40-char SHA of the commit.
        </ResponseField>
        
        <ResponseField name="message" type="string" required>
          Commit subject line (first line of commit message).
        </ResponseField>
        
        <ResponseField name="author" type="string" required>
          Commit author name.
        </ResponseField>
        
        <ResponseField name="date" type="string" required>
          ISO-8601 timestamp of commit author date.
        </ResponseField>
        
        <ResponseField name="taskId" type="string | null" required>
          Task ID from `Task:` trailer (e.g., `"TASK-F0-003"`). `null` if no Task trailer.
        </ResponseField>
        
        <ResponseField name="refIds" type="array" required>
          Artifact IDs from `Refs:` trailer.
        </ResponseField>
      </Expandable>
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="relationships" type="array" required>
  Array of all relationships between artifacts.
  
  <Expandable title="Relationship object">
    <ResponseField name="source" type="string" required>
      ID of the referencing artifact.
    </ResponseField>
    
    <ResponseField name="target" type="string" required>
      ID of the referenced artifact.
    </ResponseField>
    
    <ResponseField name="type" type="string" required>
      Relationship type:
      - `implements` - UC implements REQ
      - `orchestrates` - WF orchestrates API
      - `verifies` - BDD verifies REQ/UC
      - `guarantees` - INV guarantees domain rule
      - `decides` - ADR decides architecture
      - `decomposes` - TASK decomposes FASE
      - `implemented-by` - Task Refs field
      - `implemented-by-code` - Code references artifact
      - `tested-by` - Test references artifact
      - `implemented-by-commit` - Commit references artifact
      - `inferred-implements` - **v4 only.** Inferred codeâ†’artifact link
      - `reads-from` - FASE reads spec
      - `traces-to` - Generic cross-reference
    </ResponseField>
    
    <ResponseField name="sourceFile" type="string" required>
      File where the reference was found.
    </ResponseField>
    
    <ResponseField name="line" type="number" required>
      Line number of the reference.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="statistics" type="object" required>
  Aggregate statistics about the graph.
  
  <Expandable title="Statistics object">
    <ResponseField name="totalArtifacts" type="number" required>
      Total count of all artifacts.
    </ResponseField>
    
    <ResponseField name="byType" type="object" required>
      Count per artifact type (e.g., `{"REQ": 330, "UC": 41}`).
    </ResponseField>
    
    <ResponseField name="totalRelationships" type="number" required>
      Total count of all relationships.
    </ResponseField>
    
    <ResponseField name="traceabilityCoverage" type="object" required>
      Coverage metrics as `{count, total, percentage}` objects:
      - `reqsWithUCs` - REQs with at least one UC
      - `reqsWithBDD` - REQs with at least one BDD
      - `reqsWithTasks` - REQs with at least one TASK
      - `reqsWithCode` - REQs with at least one codeRef
      - `reqsWithTests` - REQs with at least one testRef
      - `reqsWithCommits` - REQs with at least one commitRef
    </ResponseField>
    
    <ResponseField name="orphans" type="array" required>
      Artifact IDs with no incoming references.
    </ResponseField>
    
    <ResponseField name="brokenReferences" type="array" required>
      Array of `{ref, referencedIn, line}` objects for references to undefined artifacts.
    </ResponseField>
    
    <ResponseField name="codeStats" type="object" required>
      Code scanning statistics: `{totalFiles, totalSymbols, symbolsWithRefs}`.
    </ResponseField>
    
    <ResponseField name="testStats" type="object" required>
      Test scanning statistics: `{totalTestFiles, totalTests, testsWithRefs}`.
    </ResponseField>
    
    <ResponseField name="commitStats" type="object" required>
      Commit scanning statistics: `{totalCommits, commitsWithRefs, commitsWithTasks, uniqueTasksCovered}`.
    </ResponseField>
    
    <ResponseField name="classificationStats" type="object" required>
      Classification breakdown: `{byDomain, byLayer, byCategory}` with counts.
    </ResponseField>
    
    <ResponseField name="adoptionStats" type="object | null">
      **v3 only.** Adoption score, grade, findings counts, alignment percentage. `null` if no onboarding data.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="adoption" type="object">
  **v3 only.** Onboarding skill data. Defaults to `{"present": false}` when no onboarding data exists.
  
  <Expandable title="Adoption object">
    <ResponseField name="present" type="boolean" required>
      Whether any adoption/onboarding data exists.
    </ResponseField>
    
    <ResponseField name="onboarding" type="object">
      Data from `onboarding/ONBOARDING-REPORT.md`.
    </ResponseField>
    
    <ResponseField name="reverseEngineering" type="object">
      Data from `findings/FINDINGS-REPORT.md` and `reverse-engineering/INVENTORY.md`.
    </ResponseField>
    
    <ResponseField name="reconciliation" type="object">
      Data from `reconciliation/RECONCILIATION-REPORT.md`.
    </ResponseField>
    
    <ResponseField name="import" type="object">
      Data from `import/IMPORT-REPORT.md`.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="codeIntelligence" type="object">
  **v4 only.** Code intelligence block from `/sdd:code-index`. Absent by default.
  
  <Expandable title="Code intelligence object">
    <ResponseField name="indexed" type="boolean" required>
      Whether code has been indexed.
    </ResponseField>
    
    <ResponseField name="indexedAt" type="string" required>
      ISO-8601 timestamp of last index generation.
    </ResponseField>
    
    <ResponseField name="engine" type="string" required>
      Analysis engine: `"gitnexus"` or `"regex-lite"`.
    </ResponseField>
    
    <ResponseField name="engineVersion" type="string" required>
      Engine version string.
    </ResponseField>
    
    <ResponseField name="symbols" type="array" required>
      Symbol table with artifact refs, call graph participation, and community clustering.
    </ResponseField>
    
    <ResponseField name="callGraph" type="array" required>
      Call relationships between symbols.
    </ResponseField>
    
    <ResponseField name="processes" type="array" required>
      Detected execution flows.
    </ResponseField>
    
    <ResponseField name="stats" type="object" required>
      Aggregate statistics: `{totalSymbols, symbolsWithRefs, symbolsWithInferredRefs, uncoveredSymbols, totalProcesses, processesWithRefs}`.
    </ResponseField>
  </Expandable>
</ResponseField>

## Usage

### Generation

Generate the graph with the dashboard skill:

```bash
/sdd:dashboard
```

This scans all pipeline directories and produces both `traceability-graph.json` and `index.html`.

### Consumption

**MCP Server**: The graph is loaded by the MCP server at `server/src/graph-loader.ts` with file watching and graceful degradation.

**Dashboard**: The HTML dashboard reads the graph to render the interactive UI.

**Context Hook**: The `sdd-augment-hook.js` uses the graph to inject traceability context into file operations.
